1. Our Scenario

We have two devices that need to communicate directly with each other — for example, to access a server running on one device from the other:

Device A — iPhone

  • Connected via 5G mobile network
  • Private IP assigned by carrier: 10.48.12.97
  • Sits behind the carrier's NAT (CGNAT)
  • Running a service (e.g. HTTP server) on port 8080

Device B — MacBook

  • Connected via office/home LAN
  • Private IP assigned by router: 192.168.1.42
  • Sits behind the office/home NAT router
  • Running a service (e.g. SSH server) on port 22

Goal: The iPhone wants to connect to the MacBook's SSH server, and the MacBook wants to reach the iPhone's HTTP server. Neither can — because both devices sit behind NAT.

2. The Problem

IPv4 only has ~4.3 billion addresses — not enough for every device. So devices use private IPs (e.g. 10.x.x.x, 192.168.x.x) inside their local network and share a single public IP via the router. Network Address Translation (NAT) handles this: it rewrites outgoing packets from the private IP to the router's public IP and remembers the mapping so replies find their way back. This works transparently for outgoing connections — but creates a fundamental problem for incoming ones:

The fundamental problem: A NAT only forwards incoming packets if they match an existing mapping — and mappings are only created by outgoing connections from the device behind the NAT. If nobody inside has talked to you first, your packets are silently dropped.
Important nuance: Native public IPv6 can remove much of this reachability problem because devices can again have globally routable addresses. In practice, however, many deployments are still IPv4-only, dual-stack with restrictive firewalls, or only partially IPv6-capable. NAT traversal and relay fallback therefore remain highly relevant.

2.1 NAT Types

Not all NATs behave the same way. The classic "cone vs. symmetric" labels are a useful teaching model; the more precise RFC 4787 language talks about endpoint-independent vs. endpoint-dependent mapping and filtering. Either way, the behavior determines how hard it is to establish a direct P2P connection:

NAT TypeMapping BehaviourWho Can Send Packets In?Difficulty
Full Cone Same external port for all destinations Anyone who knows the mapped port Easy
Address-Restricted Cone Same external port for all destinations Only hosts the internal device has contacted Easy
Port-Restricted Cone Same external port for all destinations Only the exact host:port pair previously contacted Medium
Symmetric NAT Different external port per destination Only the exact host:port for that specific mapping Hard

The critical difference: many easier NATs keep the public mapping stable enough that endpoint discovery remains useful. Harder, endpoint-dependent NATs may create different mappings per destination, which makes simple STUN discovery much less useful for the actual peer connection.

CGNAT — Carrier-Grade NAT

Mobile carriers add another layer with CGNAT. Your iPhone may traverse two NATs before reaching the Internet:

iPhone Carrier Internal NAT Carrier Public NAT Internet 10.48.12.97 ──▶ 100.64.5.18 ──▶ 85.195.22.1 ──▶ ... private CGNAT range (shared) public (shared by thousands of users) The 100.64.0.0/10 range is reserved specifically for CGNAT. You cannot configure port forwarding on your carrier's NAT equipment. The public IP is shared with thousands of other subscribers.

2.2 NAT Traversal Challenges

Establishing direct communication between two devices behind NAT is hard for several compounding reasons:

No Inbound Path

NAT only creates mappings for outgoing traffic. There is no way to send an unsolicited packet to a device behind NAT — the NAT has no table entry and silently drops it.

Unknown Addresses

Neither device knows the other's public IP:port. Even the device itself doesn't know its own public mapping — it only sees its private address.

Symmetric NAT Randomness

On Symmetric NAT (common on 5G), the external port changes per destination. Discovering the port via a STUN server tells you the port for that server, not for your peer.

Mapping Expiry

NAT mappings expire after inactivity (typically 30-120 seconds for UDP). Connections must send keepalive packets or the "hole" closes and communication drops.

Mobile Network Instability

5G devices change IP addresses during cell handovers. The entire NAT mapping can be invalidated mid-session, requiring re-establishment of the connection.

UDP Blocking

Some networks (enterprise firewalls, restrictive carriers) block or throttle UDP traffic entirely. For the mesh and overlay approaches discussed here, that usually eliminates the direct path and forces relay or alternate transport.

2.3 NAT Traversal Techniques

Most modern UDP-based P2P systems use variations of three core building blocks, often combined via ICE:

STUN Session Traversal Utilities for NAT

A client contacts a public STUN server to discover its own public IP:port mapping. Both peers exchange these "reflexive candidates" and attempt direct UDP hole-punching.

Zero relay cost — fully P2P once connected
Often struggles with hard / endpoint-dependent NATs and may still need relay fallback

TURN Traversal Using Relays around NAT

Fallback relay server that forwards all traffic when direct connection fails. Guarantees connectivity but adds latency and bandwidth cost.

Works through any NAT type, even Symmetric
All traffic relayed — adds latency + server cost

ICE Interactive Connectivity Establishment

Framework that orchestrates STUN and TURN. Gathers candidate paths, tests connectivity in priority order, and picks the best working path.

Graceful fallback when paired with relay candidates
Initial connection setup is more complex and can be slower than a fixed relay

Birthday Attack / Port Prediction

Advanced family of techniques for hard NAT cases: peers probe predicted port ranges or try additional coordination strategies, hoping to create a direct path without relaying. Public implementation details and real-world success rates vary by product and network.

Can improve direct-connect odds in otherwise difficult NAT scenarios
Probabilistic — not guaranteed, and highly implementation-dependent

3. Existing Solutions

Since neither device can directly reach the other, all solutions rely on the same core idea: use a third party that both devices CAN reach — a server with a public IP. The solutions differ in how much they rely on that server, and at which network layer they operate.

3.1 VPN (Network Layer)

The simplest approach: both devices connect outward to a VPN server in the cloud. Since outgoing connections work through NAT, both can establish a tunnel to the server. The server then routes traffic between them.

┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │ outgoing │ VPN Server │ outgoing │ MacBook │ │ 10.48.12.97 │────tunnel──────▶│ 203.0.113.50 │◀──────tunnel────│ 192.168.1.42 │ │ │ (works!) │ (public IP) │ (works!) │ │ │ VPN IP: │ │ │ │ VPN IP: │ │ 10.8.0.2 │◀───encrypted───▶│ Routes between │◀───encrypted───▶│ 10.8.0.3 │ └──────────────┘ traffic │ VPN clients │ traffic └──────────────┘ └──────────────────┘ ✓ Both connections are OUTGOING → both pass through NAT successfully The VPN server acts as a permanent relay for all traffic.

Pros

  • Simple, well-understood model
  • Works through any NAT type (outgoing connections always work)
  • Full L3 IP connectivity between clients
  • Mature tooling (OpenVPN, WireGuard, IPsec)

Cons

  • Network layer (L3) — usually requires a TUN/TAP interface, admin/root privileges, or a system-level VPN profile; some products also offer userspace modes, but with trade-offs
  • All traffic relayed through server — adds latency
  • Server bandwidth = bottleneck, cost scales with traffic
  • Traffic is usually decrypted and re-routed at the VPN server unless you add application-level end-to-end encryption
  • Requires provisioning and maintaining a server

Examples: OpenVPN, WireGuard (client-server mode), IPsec/IKEv2.

3.2 Overlay Networks (Mesh Overlays)

Overlay networks improve on the basic VPN by adding coordination and NAT traversal. They usually combine a control plane for identity, discovery, and policy with an L2 or L3 tunnel data plane. After setup, they attempt a direct peer-to-peer connection between devices; if direct connection fails, they fall back to relaying.

── Phase 1: Discovery (via coordination server) ── ┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │ outgoing │ Coordination │ outgoing │ MacBook │ │ │───registers────▶│ Server │◀───registers────│ │ │ │◀───receives ────│ "MacBook is at │────receives───▶ │ │ │ │ peer info │ 85.195.44.8 │ peer info │ │ └──────────────┘ └──────────────────┘ └──────────────┘ ── Phase 2: Direct connection attempt (NAT hole punching) ── ┌──────────────┐ ┌──────────────┐ │ iPhone │◀═══════ encrypted traffic flows directly ══════════▶│ MacBook │ └──────────────┘ (no server in the middle) └──────────────┘ ── Fallback: If direct connection fails ── ┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │═══encrypted════▶│ Relay Server │◀═══encrypted═══│ MacBook │ │ │◀══traffic═══════│ (DERP / Root) │════traffic════▶ │ │ └──────────────┘ └──────────────────┘ └──────────────┘

Pros

  • Overlay model — coordination and policy are often user-space, while the data plane may still use kernel or system VPN integration depending on platform
  • Direct P2P when available — lower latency and less persistent server bottleneck
  • Peer traffic is transport-encrypted end-to-end; relays can often forward ciphertext without seeing payloads
  • Server only handles tiny control messages
  • Automatic NAT traversal, handles IP changes
  • Zero-config setup (install app + join network)

Cons

  • Relies on coordination server for initial setup
  • Direct connection is not guaranteed; outcome depends on NAT behavior, IPv6 availability, and UDP reachability
  • More complex operational model than a fixed client-server VPN
  • Vendor dependency (unless self-hosted)

4. Technical Approaches

This section explains how the solutions from the previous section work under the hood — the two fundamental communication models and the protocols they use.

4.1 Relay Based Communication

In the relay model, both devices establish outgoing tunnels to a central server. The server forwards all traffic between them. This always works because outgoing connections pass through any NAT.

┌──────────────┐ ┌──────────────────┐ ┌──────────────┐ │ iPhone │ outgoing │ Relay Server │ outgoing │ MacBook │ │ │════tunnel══════▶│ │◀══════tunnel════│ │ │ │◀═══encrypted═══▶│ Forwards all │◀═══encrypted═══▶│ │ └──────────────┘ traffic │ traffic between │ traffic └──────────────┘ │ the two clients │ └──────────────────┘
1

iPhone opens an outgoing encrypted tunnel to relay server → Carrier NAT creates mapping → connected.

2

MacBook opens an outgoing encrypted tunnel to same server → Office NAT creates mapping → connected.

3

Server assigns virtual IPs and forwards packets between the two tunnels.

Trade-off: Reliability is close to universal because both sides make outbound connections, but the server stays on the data path — adding latency, bandwidth cost, and a single point of failure. Unless the application adds its own end-to-end encryption, the relay or VPN server can inspect forwarded traffic after tunnel termination.

4.2 Direct P2P Communication

In the direct P2P model, a coordination server introduces the peers, then they establish a direct connection via UDP hole punching. The trick: both devices simultaneously send packets to each other's discovered public endpoints, creating NAT mappings from both sides.

── Step 1: Both peers send UDP packets simultaneously ── iPhone sends UDP to 85.195.44.8:51820 (MacBook's NAT public side) → Carrier NAT creates mapping: 10.48.12.97:5000 ↔ 85.195.22.1:43210 → 85.195.44.8:51820 → Packet arrives at MacBook's NAT... → MacBook's NAT has no mapping yet → packet dropped (but that's OK!) MacBook sends UDP to 85.195.22.1:43210 (iPhone's NAT public side) → Office NAT creates mapping: 192.168.1.42:51820 ↔ 85.195.44.8:51820 → 85.195.22.1:43210 → Packet arrives at iPhone's Carrier NAT... → Carrier NAT HAS a mapping for 85.195.44.8:51820 → ✓ PACKET FORWARDED to iPhone! ── Step 2: Now the "hole" is punched in BOTH NATs ── iPhone sends another UDP to MacBook → Arrives at MacBook's NAT, which now has a mapping for iPhone's address → ✓ FORWARDED to MacBook! Both NATs now have active mappings → bidirectional direct communication!
Why UDP and not TCP? UDP is connectionless — you can send a packet without a handshake. TCP requires a three-way handshake (SYN → SYN-ACK → ACK) which must complete in order. TCP simultaneous-open exists in theory, but is far less reliable across consumer NATs and is rarely the practical choice for mesh overlays. For the approaches discussed here, practical direct traversal is overwhelmingly UDP-based.

5. Summary & Recommendation

The NAT problem in one sentence: NAT only allows replies to connections started from the inside — so if both devices are behind NAT, neither can start a connection to the other. Overlay networks solve this by using a public coordination server for peer discovery, then establishing direct encrypted P2P tunnels via UDP hole punching.

5.1 Reference Case: iPhone 5G ↔ MacBook LAN

If direct fails → traffic relays through nearest DERP server 5. Background: continuously re-attempts direct connection

In Swiss mobile networks, CGNAT is common enough that you should expect a mix of direct and relayed paths rather than guaranteed P2P. In practice, a product like Tailscale usually converges quickly to either a direct WireGuard path or a DERP relay. The main user-visible benefit is stable addressing, automatic recovery, and seamless fallback — not a guarantee that every session will be direct.

5.3 Outlook: QUIC, MASQUE & Modern Tunneling

While WireGuard-based overlay networks remain the most practical choice today, the tunneling landscape is also moving toward QUIC-based transports and HTTP/3 proxying:

QUIC The New Transport

HTTP/3 adoption is growing, and Cloudflare reported that 21% of requests on its network used HTTP/3 in 2025. QUIC's connection migration can help sessions survive address changes, which is valuable on mobile networks. Because QUIC often rides on the same UDP/443 path as modern web traffic, it can also be easier to accommodate in enterprise environments than custom tunnel protocols.

MASQUE IETF Standard Tunneling

MASQUE builds on HTTP over QUIC to standardize proxy-style tunneling. Today the key RFCs cover UDP proxying (RFC 9298) and IP proxying (RFC 9484). MASQUE is important for relay and proxy architectures, but it is not the same thing as generic peer-to-peer NAT traversal.

What This Means

Some future overlay systems may adopt QUIC-based transports or MASQUE-compatible relay modes where that improves firewall compatibility or mobility. That would complement today's WireGuard meshes, not automatically replace the need for coordination, policy, identity, and fallback relays.

For now: Tailscale/Headscale with WireGuard remains the pragmatic choice for this use case. Keep QUIC/MASQUE on the radar as complementary transport and relay technology, not as a proven drop-in replacement for today's mesh overlays.
